Backup & Restore

The backup system protects the Filer metadata (LevelDB state) by periodically archiving it to the dashboard server. File content stored on volume servers is not included — only the directory structure, attributes, and chunk-to-volume mappings are backed up.

Scope of Backup

What IS Backed Up

  • Filer LevelDB metadata — /data/dc03/filer/filerldb2/
  • Directory tree structure and file entries
  • File attributes (size, modification time, mode, owner)
  • Chunk-to-volume-ID mappings (the "needle map")
  • Filer store configuration and IAM data

What Is NOT Backed Up

  • Actual file content (stored on volume servers as .idx / .dat pairs)
  • Volume index files and data files
  • S3 object payloads (these live on volume servers)
Recovery strategy: Volume data is protected by replication (001 — two copies per volume). To recover from a total volume-server failure, re-replicate from surviving copies. The filer metadata backup ensures you can reconstruct the directory tree and find which chunks belong to which files.

How Backup Works

Step 1 — SSH to Filer

The dashboard server SSH's into each filer node as root using the configured SSH key. Both HA filers are backed up individually:

Filer NodeSSH TargetLevelDB Path
Primary root@10.10.95.102 /data/dc03/filer/filerldb2
Secondary root@10.10.95.104 /data/dc03/filer/filerldb2

Step 2 — Tar Archive

The filer compresses its LevelDB directory into a tarball:

tar -czf /tmp/filer-backup-20260717-120000.tar.gz \
    -C /data/dc03/filer filerldb2

Compression reduces the typical LevelDB size from ~500 MB to ~80 MB.

Step 3 — SFTP Download

The tarball is transferred via SFTP from the filer to the dashboard server:

Destination: /srv/seaweed-backups/
Filename:    filer-102_20260717-120000.tar.gz

After a successful transfer the remote tarball is cleaned up.

Step 4 — Retention

Backups older than 30 days are automatically pruned. The retention window is configurable in runtime settings.

Backup frequency: Manual trigger only — use New Backup or Sync Now buttons in the UI.

Restore Process

Step 1 — Select a Backup Snapshot

Choose a backup from the list. Only backups with uploaded status can be used for restore. Backups marked missing or failed are not restorable.

Step 2 — SFTP Upload to Filer

The backup tarball is pushed back to the target filer:

scp /srv/seaweed-backups/filer-102_20260717-120000.tar.gz \
    root@10.10.95.102:/tmp/

Step 3 — Extract into LevelDB Directory

# On the filer node (auto-executed by restore):
systemctl stop seaweed-filer
rm -rf /data/dc03/filer/filerldb2
tar -xzf /tmp/filer-102_20260717-120000.tar.gz -C /data/dc03/filer/
systemctl start seaweed-filer

Step 4 — Verify Filer Health

The dashboard checks that the filer comes back online and responds to GET /cluster/filer before marking the restore as complete.

Downtime required: The filer must be stopped during restore. S3 and filer operations are unavailable for the duration (typically 1–3 minutes).

Encryption Compatibility

The backup archive itself is not encrypted — it is a raw tarball of the LevelDB data. If encryption is required, the following approaches are compatible:

Encryption LayerCompatibilityNotes
Volume Encryption compatible SeaweedFS volume-level encryption (at rest) does not affect filer metadata
LUKS (disk) compatible Full-disk encryption on the filer node is transparent to the backup process
Transport (SSH) enforced SFTP/SCP transfers are encrypted via the SSH tunnel
Rest Encryption manual Encrypt tarballs with GPG before storing if needed

API Key Requirement

All backup and restore operations require an API key passed in the X-API-Key header: 

X-API-Key: bkp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

API keys use the bkp_ prefix convention. Configure the key in .env as BACKUP_API_KEY. The dashboard UI automatically includes this header for all backup operations.

Guard the API key: Anyone with a valid backup API key can download filer metadata and trigger restore operations. Store it securely and rotate periodically.

Retention Policy

Backups are kept for 30 days by default. A cron-style cleanup job runs daily and removes any archive older than the retention threshold from /srv/seaweed-backups/.

The retention period is configurable via runtime_settings: 

Key:   backup_retention_days
Value: 30 (default)

Backup Actions

New Backup

Creates a fresh backup by SSH-ing to each filer, archiving the LevelDB, and downloading the tarball. This is a full backup — incremental/differential backups are not supported.

admin only

Sync Now

Reconciles the backup list with what's actually on disk at /srv/seaweed-backups/. Useful after manually adding or removing backup files, or if the dashboard backup metadata gets out of sync.

admin only

Backup Status Indicators

StatusMeaningRestore Available
uploaded Backup completed successfully; tarball present on dashboard server yes
partial One filer backed up but the other failed or is unreachable partial
failed Backup operation did not complete (SSH error, disk full, tar failure) no
missing Tarball metadata exists but the file is absent from disk no
running Backup is currently in progress (SSH + tar + download) wait
Restore button behavior: The Restore button is disabled (greyed out) for backups with missing or failed status. It is enabled only when at least one filer's backup is marked uploaded.

Disaster Recovery Checklist

  1. Identify the latest backup with uploaded status
  2. Verify the backup file exists: ls -lh /srv/seaweed-backups/
  3. Click Restore on the target backup entry
  4. Monitor the restore progress in the UI
  5. Once complete, verify filer is healthy via GET /cluster/filer
  6. Spot-check a few known file paths to confirm directory tree integrity
