Users & Roles

The dashboard uses a role-based access control (RBAC) system with three built-in roles. Users authenticate via session-based login (bcrypt password hashing) and every user automatically receives S3 credentials for direct SeaweedFS S3 gateway access.

Seeded accounts: admin (full access) and viewer (read-only). Credentials are set via ADMIN_USER/ADMIN_PASSWORD and READONLY_USER/READONLY_PASSWORD in .env.

Role-Based Access Control

Permissions are defined in rbac.json at the project root. Each role maps to a list of granular permission strings in domain:action format:

admin — Full System Access

Complete control over every aspect of the system, including user management and settings.

operator — Operational Management

Full cluster operations without the ability to modify users or system settings.

Identical to admin permissions minus: users:read, users:write, settings:read, settings:write, collections:delete, audit:read.

viewer — Read-Only Monitoring

Can view all dashboards and data but cannot perform any write operations.

Limited to read-only permissions across all domains plus sse:read. The default role for new users when no role is specified.

User Management (CRUD)

Create User

Requires: Admin role. The create form includes:

  • Username, first name, last name, email — all required
  • Phone — optional
  • Role — dropdown: admin, operator, viewer (defaults to default_role from rbac.json)
  • Password — set at creation time
  • S3 permission — readwrite (default) or read
  • Create S3 bucket — checkbox to auto-create user-{username} bucket
POST /api/users
{
  "username": "operator1",
  "password": "secure-password",
  "firstname": "Jane",
  "lastname": "Smith",
  "email": "jane@example.com",
  "role": "operator",
  "create_bucket": true,
  "s3_permission": "readwrite"
}

Response includes generated S3 credentials and bucket creation status:

{
  "username": "operator1",
  "role": "operator",
  "enabled": true,
  "s3_access_key": "AK",
  "s3_secret_key": "sk_",
  "bucket_created": true
}

Update User

Requires: Admin role. All fields are optional — only provided fields are updated:

PUT /api/users/{user_id}
{
  "firstname": "Janet",
  "role": "admin",
  "enabled": false
}

Enabling/disabling a user sets enabled to 1 or 0. Disabled users cannot log in.

Delete User

Requires: Admin role. The admin user cannot be deleted.

DELETE /api/users/{user_id}

{"ok": true}

Any user creation, update, or deletion triggers an asynchronous S3 gateway sync across all configured gateways.

S3 Bucket Auto-Creation

Every user automatically receives S3 access credentials (s3_access_key, s3_secret_key) generated at creation time:

  • Access key format: AK + 20 hex characters (secrets.token_hex(10))
  • Secret key: 40 hex characters (secrets.token_hex(20))

When create_bucket: true is passed, a bucket named user-{username} is created in the filer via POST /{bucket}?op=mkdir. This can also be triggered later by the user themselves: 

POST /api/users/me/bucket

{"bucket": "user-operator1", "ok": true}

Password Management

Self-Service Password Change

Any authenticated user can change their own password. The current password must be verified first:

POST /api/users/me/password
{
  "current_password": "old-password",
  "new_password": "new-secure-password"
}

The current password is verified against the stored bcrypt hash. On success, the new password is hashed with bcrypt and stored. The session remains active.

Admin Password Reset

There is no dedicated "reset password" endpoint. An admin updates a user's password by creating a new user with the same username (the UNIQUE constraint will fail) or directly via the database. A password reset UI is planned for a future release.

User Profile Management

Each user can view and update their own profile information:

GET /api/users/me/profile

{
  "username": "operator1",
  "firstname": "Jane",
  "lastname": "Smith",
  "email": "jane@example.com",
  "phone": "+97699112233",
  "role": "operator",
  "s3_access_key": "AK",
  "created_at": "2026-07-15 10:30:00"
}
PUT /api/users/me/profile
{
  "firstname": "Janet",
  "email": "janet@newdomain.com",
  "phone": "+97699887766"
}

Updatable fields: firstname, lastname, email, phone. Role and username cannot be changed by the user.

The admin system user cannot be deleted (returns 400). Always ensure at least one admin user exists in the database.

Schema

CREATE TABLE users (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    username TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    firstname TEXT NOT NULL DEFAULT '',
    lastname TEXT NOT NULL DEFAULT '',
    email TEXT NOT NULL DEFAULT '',
    phone TEXT NOT NULL DEFAULT '',
    role TEXT NOT NULL DEFAULT 'viewer',
    enabled INTEGER NOT NULL DEFAULT 1,
    s3_access_key TEXT NOT NULL DEFAULT '',
    s3_secret_key TEXT NOT NULL DEFAULT '',
    s3_permission TEXT NOT NULL DEFAULT 'readwrite',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
