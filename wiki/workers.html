Workers

The worker system manages background job execution within the SeaweedFS cluster. Jobs run on the filer nodes and handle tasks like replication repair, volume balancing, garbage collection, and custom processing pipelines.

Worker Status Monitoring

The dashboard polls each filer's worker endpoint (/cluster/workers) to track active workers and their health.

Health Indicators

StateIndicatorDescription
Healthy active Worker process is running and responding to heartbeat checks
Degraded lagging Worker is alive but job queue is growing faster than processing rate
Unhealthy unreachable Worker has not responded within the heartbeat timeout window
Stopped idle Worker is registered but has no active jobs and is waiting

View worker status via the API:

GET /api/workers/status

Job History & Lifecycle

Every job passes through a defined lifecycle with four terminal states. The job history table retains records for completed and failed jobs.

Job States

StateBadgeDescription
Pending pending Job is queued and waiting for an available worker slot
Running running Job is actively executing on a worker node
Success success Job completed without errors
Failed failed Job terminated with an unrecoverable error

List all jobs:

GET /api/workers/jobs

Triggering Jobs

Detect Jobs

Scans the cluster for work that needs to be done and enqueues the appropriate jobs. This is a read-only reconnaissance operation — it identifies tasks but does not execute them.

What it scans for:

  • Volumes with replica count below the configured replication factor
  • Uneven volume distribution across servers
  • Orphaned volume IDs referenced by filer but missing from volume servers
  • Garbage collections that are overdue
POST /api/workers/jobs/detect

admin only

The detect operation returns a summary of enqueued jobs. Each job type has a configurable concurrency limit so that a single detection run won't overwhelm the cluster.

Execute Jobs

Triggers execution of all pending jobs in the queue. Workers pick up jobs in FIFO order. A job that fails may be retried up to its configured max_retries count before being marked as failed. 

POST /api/workers/jobs/execute

admin only

Idempotency: Most job types are designed to be idempotent. Running the same job twice should produce the same result without side effects. Exceptions are noted in the job type documentation below.

Workflow

A typical maintenance workflow using the worker system:

  1. Click Detect Jobs — the dashboard scans the cluster and enqueues any needed work
  2. Review the pending job list — check types, estimated duration, and concurrency limits
  3. Click Execute Jobs — workers begin processing the queue
  4. Monitor progress in the job list — each job shows progress_pct
  5. Investigate any failed jobs — check the error field for details
  6. Optionally execute again to retry failed jobs (up to their max_retries)
Best practice: Run detection and execution during off-peak hours. Jobs like volume_balance and volume_fix_replication transfer significant data between nodes and can impact S3/filer performance.
