Volume Management

SeaweedFS volumes are the fundamental storage units — each a 30 GB block on a volume server's disk. The Volume Management page lets you view, grow, clean up, and organize volumes across the entire cluster.

Volume List

The volume list is a searchable, sortable, filterable table showing every volume in the cluster. Key columns include volume ID, collection, size, server location, replication, and status.

API Endpoints

GET  /api/volumes              # List all volumes
GET  /api/volumes/{volume_id}  # Get volume detail

Volume Detail Drawer

Click any volume row to open a side drawer with detailed information:

  • Full metadata — ID, collection, disk type, file count, delete count
  • Storage breakdown — data size vs. garbage size, compaction status
  • Replication info — type, number of replicas, replication status per replica
  • TTL status — expiration time, remaining duration
  • Server context — host node IP, hostname, disk usage of the parent server
Replication type 001: In the dc03 cluster, volumes use 001 replication — the primary copy plus one replica on a different node. Total: 2 copies of each volume. The detail drawer shows the location of each replica.

Volume Grow

Creating new volumes increases total cluster capacity. The grow operation assigns one or more new 30 GB volumes to available volume servers.

Grow Operation

1
Open the grow modal Click "Grow Volume" button — admin only
2
Configure parameters Set count, replication, collection, TTL, data center, rack
3
Validate node capacity Backend checks each target node's volume limit — blocks grow if any node is full
4
Execute & confirm Master assigns volumes, result shown in toast notification

Grow Request Parameters

ParameterDefaultDescription
count1Number of volumes to create
replication001Replication type (dc03 default: 001 = 2 copies)
collection—Assign to a named collection (optional)
ttl—Auto-expire after duration (e.g., 7d, 30d)
data_centerdc03Target datacenter
rackrack2Target rack

API Endpoint

POST /api/volumes/grow
Content-Type: application/json

{
  "count": 2,
  "replication": "001",
  "collection": "images",
  "ttl": "30d",
  "data_center": "dc03",
  "rack": "rack2"
}
Admin only: Volume grow requires admin authentication. Read-only users cannot create volumes. Growth is automatically blocked if any target node has reached its configured volume limit.

Vacuum (Garbage Collection)

When files are deleted or overwritten in SeaweedFS, the old data remains on disk as "garbage" until vacuum compaction reclaims that space. The vacuum trigger initiates garbage collection across all or selected volumes.

When to Vacuum

  • After bulk deletions — large cleanup operations leave significant garbage
  • Low disk space — if free space is tight, vacuuming can recover gigabytes
  • Periodic maintenance — schedule during low-traffic windows

How It Works

1
Trigger Admin clicks "Vacuum" or sends POST /api/volumes/vacuum
2
Compaction Volume server rewrites each volume, excluding deleted/tombstoned records
3
Commit New compacted files replace old ones; disk space is reclaimed

API Endpoint

POST /api/volumes/vacuum
Content-Type: application/json

{
  "garbage_threshold": 0.3
}

# Optionally target a specific collection:
{
  "collection": "images"
}
Performance note: Vacuum rewrites volume files on disk. On HDDs with large volumes, this can take minutes and impact I/O. Consider running vacuum during maintenance windows. The garbage_threshold parameter (0.0–1.0) controls the minimum garbage ratio required to trigger vacuum on a volume — set higher to only compact volumes with significant garbage.

Collections Management

Collections are logical groupings of volumes — useful for organizing data by purpose (e.g., "images", "backups", "logs"). The Collections page lists all collections with volume counts and supports deletion.

API Endpoints

GET    /api/collections              # List all collections with volume counts
DELETE /api/collections/{name}       # Delete a collection (admin)
Deleting a collection removes all volumes in it. This is irreversible. The delete endpoint requires admin authentication and shows a confirmation dialog before execution.

Volume Status Indicators

Each volume displays status tags that communicate its current state:

IndicatorTagMeaning
HealthyHealthyNormal operation, writable
ReadOnlyReadOnlyVolume cannot accept writes — usually disk full or filesystem error
CompactingCompactingVacuum/garbage collection in progress
FullFullVolume has reached its 30 GB capacity limit
TTL ActiveTTL: 7dVolume will auto-expire and release space after the specified duration
Monitor ReadOnly volumes closely. The alert engine triggers when the number of readonly volumes exceeds the configured threshold (default: 3). A readonly volume means its disk partition may be full or experiencing errors. Investigate immediately.

Capacity Planning

Understanding how many volumes can fit on each node is essential for cluster sizing:

dc03 Capacity Formula

Node disk:        ~1,800 GB (XFS, ~1.8 TB)
Volume size:        30 GB (configured at startup)
Max volumes:        60 per node (1800 / 30)
Replication:        001 (2x overhead)
Effective cap:      30 volumes per node (after replication)
                                    or
                    ~900 GB usable per node

Cluster total:  7 nodes × ~900 GB = ~6.3 TB usable

Per-node volume limits are set on the Cluster page. When a node approaches its limit, the dashboard warns you to either grow volumes on other nodes or increase the limit if disk space allows.
