API Key Authentication Flow

API keys provide programmatic access to the SeaweedFS Dashboard for automation, CI/CD pipelines, and service-to-service communication. Keys use the X-API-Key HTTP header and bypass session/cookie-based auth entirely. Each key is assigned granular permissions at creation time, and usage is tracked (count, last used timestamp, last endpoint).

Keys are generated with the prefix bkp_ followed by 64 hex characters (secrets.token_hex(32)). The full key is shown exactly once at creation time and stored as-is (not hashed) in the api_keys table — unlike user passwords which use bcrypt.

Step-by-Step: Key Lifecycle

1
Create Key Admin navigates to API Keys page, opens modal, enters name and selects permissions via checkboxes. Backend generates "bkp_" + secrets.token_hex(32) (68 chars total). The full key is shown exactly once — after the modal closes, it's masked.
2
Service Uses Key Service/CLI sends X-API-Key: bkp_... header. AuthMiddleware validates: checks prefix bkp_, queries api_keys table for active key. Sets request.state.user = "api_key", role = "backup_admin", permissions from DB. Records usage (increment counter, set last_used_at and last_used_endpoint).
3
Permission Check For backup_admin role, require_permission() checks if the required permission string exists in request.state.permissions (from the key's DB record). Returns 403 if the permission is not in the list. For session users, delegates to rbac.json role mapping.
4
Reveal Key To view a previously created key, admin clicks the eye icon, enters their admin password in a modal. Backend verifies via bcrypt.checkpw() against the current admin's stored hash. If correct, the full key is returned. This ensures a hijacked session cannot reveal keys without the admin password.
5
Revoke Key Admin clicks delete/revoke, confirms in a Popconfirm. Backend sets is_active = 0 (soft delete). The key remains in the database for audit but all future requests return 401. The list shows a Revoked tag.
6
View Details Clicking the info icon opens a detail drawer showing: name, permissions, created_at, last_used_at, usage_count, last_used_endpoint, created_by, is_active status. The key itself is not shown in the detail view.

Usage Tracking

Every API key request triggers record_usage() which updates:

UPDATE api_keys
SET usage_count = usage_count + 1,
    last_used_at = '<ISO 8601 timestamp>',
    last_used_endpoint = '/api/backup/status'
WHERE id = <key_id>

This allows administrators to see which keys are actively used vs dormant, identify the last endpoint each key accessed, and decide whether to revoke unused keys.

Revocation

Revocation is a soft delete — the is_active column is set to 0. The key remains in the database for audit purposes but is rejected by validate_api_key():

async def validate_api_key(key: str) -> dict | None:
    db = await get_db()
    cursor = await db.execute(
        "SELECT * FROM api_keys WHERE key = ? AND is_active = 1", (key,)
    )
    row = await cursor.fetchone()
    return dict(row) if row else None

Revoked keys return 401 on any future request, remain visible in the list with a red Revoked tag, and cannot be revealed or re-activated.
