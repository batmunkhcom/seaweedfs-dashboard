Restore Flow

The restore operation takes a previously created backup (a .tar.gz archive of the filer's LevelDB metadata) and pushes it back to the filer nodes, overwriting the current filer database. This is a destructive operation — the current filer metadata is fully replaced. After restore, the filer service must be manually restarted for changes to take effect.

The filer LevelDB contains only metadata (file names, paths, sizes, permissions, directory structure). File content resides on volume servers and is not affected by this operation. After a restore and filer restart, the filer reconnects to existing volumes and rebuilds its metadata-to-volume mappings.

Step-by-Step Explanation

1
Initiate Restore User clicks "Restore" on a backup snapshot in the Backup page. A confirmation modal with a danger button is shown, warning that the filer database will be overwritten and that manual restart is required.
2
Validation Checks that backup_enabled is "true" (from runtime settings) and that the backup file exists at /srv/seaweed-backups/{name}.tar.gz. Returns 404 if the file is not found.
3
Configuration Lookup Fetches seaweedfs_filer_host and backup_filer_db_path from runtime settings.
4
Restore to Primary Filer Only Only the first filer host (filer_hosts[:1]) is used — 10.10.95.102. The archive is uploaded via SFTP, then extracted with tar xzf directly into the LevelDB directory.
5
Manual Filer Restart Critical The user must manually SSH into both filer nodes and run systemctl restart seaweed-filer. The filer reads the restored LevelDB at startup, reconnects to volume servers, and rebuilds metadata-to-volume mappings automatically.

Why Only the Primary Filer?

The restore logic restores to only the first filer host. This is because:

  1. Both filer nodes (10.10.95.102 and 10.10.95.104) are in an HA group (ha, filerGroup=ha)
  2. They synchronize metadata via gRPC peer sync on port 18888
  3. After restoring one node and restarting both, the peer sync replicates the restored data to the other node

Why Filer Restart is Required

The filer reads its LevelDB database at startup and keeps it in memory (with periodic flushes). Simply overwriting the LevelDB files on disk while the filer is running:

  1. Does not reload the in-memory data structures
  2. May cause corruption if the filer writes to the database concurrently
  3. Leaves stale in-memory state that contradicts the new on-disk state

After a restart, the filer reads the restored LevelDB from disk, loads all metadata into memory, reconnects to volume servers, and resumes normal operation.

When the filer restarts with restored metadata, it scans the volume servers, finds which volumes hold which file chunks, and rebuilds the in-memory mapping. File content stays on the volume servers — the restore only recovers the "index" (metadata).
