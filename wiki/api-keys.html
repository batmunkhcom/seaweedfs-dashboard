API Keys

The API Key system provides token-based authentication for programmatic access to the dashboard backend. Unlike session-based auth (login form + cookie), API keys are designed for automated clients, backup scripts, and service-to-service communication.

API keys are not limited to backup use cases. They support a flexible permission model covering backup, filer, S3, and worker operations.

Key Format

Every API key follows the format: 

bkp_<64-character hex token>

The bkp_ prefix is required — keys without it are rejected during validation. The token body is generated via secrets.token_hex(32) producing a cryptographically random 64-character hex string. Example: 

bkp_a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2

Permission Model

Each API key has a set of granular permissions, stored as a comma-separated string in the database. There are 8 available permission scopes:

Permission Scope Typical Use Case
backup:read Reading List backup status, view snapshots
backup:write Writing Trigger sync, create/delete snapshots
filer:read Reading Browse directories, list files
filer:write Writing Upload, delete, create directories
s3:read Reading List buckets, users, policies
s3:write Writing Create/delete buckets, manage users
workers:read Reading View worker status, job history
workers:execute Writing Trigger detect/execute jobs

The default permission set for a new key is backup:read,backup:write. Keys can have any combination of the 8 permissions.

Key Creation

Only admin users can create API keys. The creation UI presents:

  • A name field for human identification of the key
  • A permission selector — checkbox group with all 8 permission options

Creating a key via API:

curl -X POST http://localhost:8000/api/api-keys/create \
  -H "Content-Type: application/json" \
  -H "Cookie: session=..." \
  -H "X-CSRF-Token: ..." \
  -d '{
    "name": "Nightly Backup Script",
    "permissions": "backup:read,backup:write"
  }'

Response:

{
  "key": "bkp_a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2",
  "name": "Nightly Backup Script",
  "permissions": "backup:read,backup:write"
}
The full key value is returned only once — at creation time. Copy and store it securely. It cannot be retrieved later without the reveal flow.

Key Reveal

Once a key has been created, its plaintext value is not displayed in any list view. To retrieve the key again, use the reveal endpoint which requires the current user's admin password for verification: 

curl -X POST http://localhost:8000/api/api-keys/reveal \
  -H "Content-Type: application/json" \
  -H "Cookie: session=..." \
  -H "X-CSRF-Token: ..." \
  -d '{"key_id": 1, "admin_password": "your-password"}'

The password is verified against the current session user's bcrypt hash in the users table. If incorrect, the request is rejected with a 403. Successful reveals are logged.

Key Revocation

Admin users can revoke (deactivate) a key. Revocation sets is_active = 0 in the database — the key record is preserved for audit but future authentication attempts will be rejected. 

curl -X POST http://localhost:8000/api/api-keys/revoke/1 \
  -H "Cookie: session=..." \
  -H "X-CSRF-Token: ..."

Revoked keys return None from validate_api_key(), resulting in a fallback to session-based auth (or 401 if unauthenticated).

Usage Tracking

Each API key tracks three usage metrics:

Column Description
usage_count Total number of authenticated requests made with this key
last_used_at ISO 8601 timestamp of the most recent request
last_used_endpoint The URL path of the most recent request (e.g., /api/backup/status)

Usage is recorded synchronously in the auth middleware via record_usage() on every successful API key authentication. This data is visible in the key list and detail drawer in the UI.

Detail Drawer

Clicking a key row in the UI opens a detail drawer showing:

  • Key name and ID
  • Permission list (rendered as tags)
  • Created by (username of the admin who created the key)
  • Creation timestamp
  • Active/revoked status badge
  • Usage count, last used timestamp, and last endpoint

The detail endpoint returns:

GET /api/api-keys/{key_id}/detail

{
  "id": 1,
  "name": "Nightly Backup Script",
  "permissions": ["backup:read", "backup:write"],
  "created_at": "2026-07-15 10:30:00",
  "last_used_at": "2026-07-17 02:00:00",
  "is_active": 1,
  "usage_count": 47,
  "last_used_endpoint": "/api/backup/status",
  "created_by": "admin"
}

Note that permissions is stored as a CSV string in the database ("backup:read,backup:write") but returned as a JSON array in the API response for easier consumption.

Authentication Flow

API key authentication is handled in the auth middleware:

1
Client sends X-API-Key header The middleware checks for this header before falling back to session auth.
2
Prefix validation Key must start with bkp_ — rejected immediately if not.
3
Database lookup Queries api_keys WHERE key = ? AND is_active = 1.
4
Set request state request.state.user = "api_key", request.state.role = "backup_admin", request.state.permissions = [...], request.state.api_key_id = key_data["id"]
5
Record usage Increments usage_count, updates last_used_at and last_used_endpoint.
6
Permission check Protected routes use require_permission() which checks the key's explicit permission list (not RBAC roles). If the required permission is absent, 403 is returned.
API key users are assigned the special role backup_admin internally, but this role does not use RBAC — permissions are checked directly against the key's permission list. This keeps API keys independent from the user RBAC system.

Using an API Key

Include the key in the X-API-Key header of any request. No session cookie or CSRF token is needed:

curl -s http://localhost:8000/api/backup/status \
  -H "X-API-Key: bkp_a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2"

This works for any client that can set HTTP headers — shell scripts, Python httpx/requests, Go, etc.

API keys bypass CSRF protection and session timeout. Treat them like passwords. Rotate them regularly and revoke any that are compromised.

Schema

CREATE TABLE api_keys (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    key TEXT UNIQUE NOT NULL,
    name TEXT NOT NULL,
    permissions TEXT NOT NULL DEFAULT 'backup:read,backup:write',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    last_used_at DATETIME,
    is_active INTEGER NOT NULL DEFAULT 1,
    usage_count INTEGER DEFAULT 0,
    last_used_endpoint TEXT,
    created_by TEXT DEFAULT 'admin'
);
