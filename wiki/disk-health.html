Disk Health Monitoring

The Disk Health feature provides S.M.A.R.T. monitoring of physical disks across all SeaweedFS cluster nodes. It uses SSH to connect to each node, collects device and temperature data via smartctl and lsblk, and stores results in the local SQLite database for historical trending.

Disk Health is disabled by default. Set DISK_HEALTH_ENABLED=true in .env and restart the backend to activate it.

Enabling Disk Health

Add the following entries to your .env file:

DISK_HEALTH_ENABLED=true
DISK_HEALTH_SSH_USER=root
DISK_HEALTH_SSH_KEY_PATH=~/.ssh/id_rsa

After restart, the service begins scheduled scans across all nodes defined in SEAWEEDFS_MASTER_HOSTS, SEAWEEDFS_FILER_HOST, and SEAWEEDFS_VOLUME_HOSTS. Duplicate IPs are automatically deduplicated.

SSH connectivity to every cluster node is required. The private key must be accessible from the backend process and target user (default root) must accept key-based authentication.

SSH Requirements

  • Key-based authentication only — password auth is not supported
  • Default key path: ~/.ssh/id_rsa (configurable via DISK_HEALTH_SSH_KEY_PATH)
  • Default SSH user: root (configurable via DISK_HEALTH_SSH_USER)
  • SSH timeout: 10 seconds per connection attempt
  • Each node is scanned sequentially using paramiko in a thread pool executor

Verify SSH access to each node before enabling:

ssh -i ~/.ssh/id_rsa root@10.10.95.101 echo "OK"

Scan Process

The scan executes two passes on each node:

Pass 1 — S.M.A.R.T. Scan

  1. Runs lsblk --json -o NAME,SIZE,TYPE,MOUNTPOINT to discover block devices
  2. Filters for devices with type == "disk"
  3. For each disk device, runs smartctl --json -a /dev/{name}
  4. Stores the raw JSON output in the disk_health table

Pass 2 — Fallback (when no S.M.A.R.T. devices found)

  1. Re-runs lsblk --json -b for detailed block device info
  2. Issues df -B1 on mountpoints under /data for usage data
  3. Inserts a synthetic S.M.A.R.T. record with "model_name": "Virtual Disk"
  4. Skips devices < 100 GB that don't have a /data mount

Scheduled Scans

The background scan loop runs continuously. After each complete scan of all nodes, it sleeps for the configured interval before repeating.

Setting Key Default Source
disk_health_scan_interval_hours 24 runtime_settings (change via Settings UI)

The service implements these safety guarantees:

  • Singleton enforcement — only one scan loop runs at a time; starting a new instance cancels the previous
  • Restart cooldown — 300 seconds between restarts to prevent thrashing
  • Heartbeat — updates services_health table every scan with 300s TTL

Alert Thresholds

All thresholds are stored in runtime_settings and configurable via the Settings UI:

Setting Key Description Default
disk_health_temp_warn_c Temperature warning threshold 55 °C
disk_health_temp_crit_c Temperature critical threshold 65 °C
disk_health_wear_warn_pct SSD wear level warning 85 %
disk_health_realloc_warn_count Reallocated sector warning count 10

Temperature applies to both SSDs and HDDs. Wear level applies to SSDs only (via ata_smart_attributes). Reallocated sector count applies to HDDs. Alerts fire when any threshold is exceeded or if a node is unreachable.

Disk Health Page Overview

The Dashboard UI provides a dedicated Disk Health page with these sections:

Status Summary

Lists all scanned devices across the cluster — node hostname, device path, last scan timestamp. Each device shows a status badge: OK, Warning, or Critical based on the configured thresholds.

Key Metrics per Device

  • Temperature — current reading from S.M.A.R.T. data
  • Wear Level — SSD lifetime remaining (if available)
  • Reallocated Sectors — count of remapped bad sectors
  • Disk Usage — used/available from df for /data mountpoints
  • S.M.A.R.T. Status — overall pass/fail from smartctl

Per-Device Detail View

Clicking a device row opens a detail drawer showing the full S.M.A.R.T. report:

  • Device model and serial number
  • User capacity (bytes)
  • All ATA S.M.A.R.T. attributes table
  • Temperature history chart for the selected device

Historical Data Charts

The GET /api/disk-health/history/{node}/{device}?days=30 endpoint returns time-series data for temperature and wear level over the requested period (default 30 days). The frontend renders this as a line chart (Recharts) with threshold lines overlaid for warn/crit levels. 

# Fetch 30-day temperature history for a specific device
curl -s "http://localhost:8000/api/disk-health/history/10.10.95.101/sda?days=30"

Each data point includes:

  • timestamp — Unix timestamp of the scan
  • temp — Temperature in °C (from S.M.A.R.T. data)
  • wear — Wear level value (from ATA attributes table, 0 if unavailable)

Manual Scan Trigger

Administrators can trigger an immediate scan from the Disk Health page or via API: 

# Trigger an immediate scan (requires cluster:write permission)
curl -X POST http://localhost:8000/api/disk-health/scan \
  -H "Cookie: session=..." \
  -H "X-CSRF-Token: ..."

The scan runs asynchronously; the endpoint returns {"ok": true} immediately.

API Reference

GET /api/disk-health/status

Returns whether disk health is enabled and a list of scanned devices.

{
  "enabled": true,
  "devices": [
    {"node": "10.10.95.101", "device": "/dev/sda", "last_scan": 1715900000.0}
  ]
}

POST /api/disk-health/scan

Triggers an immediate scan of all nodes. Requires cluster:write permission.

{"ok": true}

GET /api/disk-health/{node}/{device}

Returns the latest S.M.A.R.T. record for a specific device.

{"node": "10.10.95.101", "device": "/dev/sda", "timestamp": 1715900000.0, "smart": {...}}

GET /api/disk-health/history/{node}/{device}?days=30

Returns time-series temperature and wear data for charting.

[{"timestamp": 1715900000.0, "temp": 42, "wear": 95}]

Schema

CREATE TABLE disk_health (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    node TEXT NOT NULL,
    device TEXT NOT NULL,
    timestamp REAL NOT NULL,
    smart_json TEXT
);

CREATE INDEX idx_disk_health_node_device ON disk_health(node, device);
CREATE INDEX idx_disk_health_ts ON disk_health(timestamp);
