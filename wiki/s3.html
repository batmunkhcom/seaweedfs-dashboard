S3 Object Store

The SeaweedFS S3 Gateway provides an S3-compatible API across four cluster nodes. This page documents bucket management, user/secret generation, IAM policy editing, and gateway endpoint details.

S3 Gateway Endpoints

Node Endpoint Status
st95102 10.10.95.102:8333 online
st395104 10.10.95.104:8333 online
st395106 10.10.95.106:8333 online
st395107 10.10.95.107:8333 online
S3-compatible clients — Use any of the four gateways. The dashboard selects the first available endpoint when proxying requests via the backend API at /api/s3/.

Example listing buckets with the AWS CLI:

aws s3 ls --endpoint-url=http://10.10.95.102:8333

S3 Buckets — CRUD

Buckets are listed, created, and deleted through the S3 management UI under S3 → Buckets. All operations proxy through the dashboard backend to the S3 Gateway.

List Buckets

Shows all existing buckets with name, creation date, and approximate object count.

GET /api/s3/buckets

Response: Array of bucket objects with Name, CreationDate.

Create Bucket

Creates a new S3 bucket. Bucket names must be DNS-compliant and unique across the cluster.

POST /api/s3/buckets
Content-Type: application/json

{ "name": "my-bucket" }

admin only

Delete Bucket

Permanently deletes a bucket. The bucket must be empty before deletion.

DELETE /api/s3/buckets/{name}

admin only

Warning: Deleting a non-empty bucket will fail. Empty all objects via the filer browser or S3 client before attempting deletion.

S3 Users & Secrets

S3 users represent IAM identities for programmatic access. Each user receives a unique Access Key and Secret Key pair generated by the Filer IAM service (embedded on each filer at port 8111).

List Users

GET /api/s3/users

Returns all registered S3 users with their access keys and creation timestamps.

Create User

POST /api/s3/users
Content-Type: application/json

{ "name": "service-account-1" }

admin only

On creation the server generates a access_key (20 chars) and secret_key (40 chars). The secret is displayed once — copy it immediately as it cannot be retrieved later.

Key Format: Access keys follow the pattern AKIA... (AWS-compatible). Secret keys are hex-encoded random bytes.

Delete User

DELETE /api/s3/users/{accessKey}

admin only

Removes the user and revokes all associated permissions. Existing signed URLs or active sessions using this identity will fail immediately.

Secret Key Security: Secrets are stored hashed in the Filer LevelDB. The dashboard never logs or caches secret keys. If a key is lost, you must create a new user — secrets cannot be recovered.

S3 Policies

Policies control access permissions for S3 users. Edit policies through the built-in JSON editor with real-time validation.

List Policies

GET /api/s3/policies

Create / Update Policy

PUT /api/s3/policies/{policyName}
Content-Type: application/json

{
  "policy": {
    "Version": "2012-10-17",
    "Statement": [
      {
        "Effect": "Allow",
        "Action": ["s3:GetObject", "s3:ListBucket"],
        "Resource": ["arn:aws:s3:::my-bucket/*"]
      }
    ]
  }
}

admin only

Policy Validation

The JSON editor validates against the AWS IAM policy schema before saving:

  • Version must be a valid date string (e.g., 2012-10-17)
  • Each Statement must have Effect (Allow or Deny)
  • Action and Resource arrays are required

Example policy granting read-only access to a single bucket:

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject"],
      "Resource": ["arn:aws:s3:::reports/*"]
    }
  ]
}

IAM Sync

The Filer IAM store is embedded in each filer at port 8111. When users or policies change on one filer, the IAM store must be synced to the peer filer in the HA group.

Trigger IAM Sync

Clicking Sync IAM triggers the filer to broadcast its IAM state to peers via the /iam/sync endpoint. 

POST /api/s3/iam/sync

admin only

IAM Sync Status

Possible status values and their meaning:

StatusMeaning
synced IAM stores on all filer peers are identical
syncing Sync in progress — changes propagating to peer filer
mismatch Filer peers disagree on IAM state — manual sync required
HA Group: Filer HA group ha runs on nodes 10.10.95.102 and 10.10.95.104. IAM sync is bidirectional between these two filers.

Testing S3 Access

Quick validation using awscli on any machine with network access:

# Configure credentials (from S3 Users page)
aws configure set aws_access_key_id AKIAxxxxxxxxxxxx
aws configure set aws_secret_access_key xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

# Test against any gateway
aws s3 ls --endpoint-url=http://10.10.95.102:8333
aws s3 mb s3://test-bucket --endpoint-url=http://10.10.95.104:8333
aws s3 cp file.txt s3://test-bucket/ --endpoint-url=http://10.10.95.106:8333

For programmatic access, configure your SDK with a custom endpoint and path-style addressing (SeaweedFS does not use virtual-hosted-style buckets).
