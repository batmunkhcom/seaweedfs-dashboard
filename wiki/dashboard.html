Dashboard Overview

The dashboard is the landing page after login. It provides a real-time, high-level view of the entire SeaweedFS cluster: key performance indicators, resource usage, and active alerts — all refreshed automatically.

KPI Stat Cards

Six stat cards sit at the top of the dashboard, showing critical metrics at a glance:

Metrics Displayed

Total FilesFiler APINumber of files tracked by the filer

Free SpaceVolume serversAvailable disk space percentage across all nodes

Real-time SSE Data Stream

The dashboard uses Server-Sent Events (SSE) for real-time updates — no polling, no stale data. A single persistent SSE connection pushes cluster stats to the browser whenever values change.

SSE Architecture

1
Frontend connects useSSE hook opens GET /api/dashboard/sse via EventSource API
2
Backend polls masters SseManager periodically fetches /api/dashboard/stats from the master API
3
Data pushed to browser Changes are emitted as SSE events with event: stats type
4
UI updates reactively Zustand store receives the event, components re-render with fresh data
Auto-reconnect: The SSE client automatically reconnects with exponential backoff (1s → 2s → 4s → 8s → max 30s) on connection loss. If Redis is configured, SSE events use pub/sub for multi-worker deployments.

Historical Charts

The SnapshotService runs in the background, polling cluster metrics at a configurable interval (default: 60 seconds) and storing them in SQLite. This time-series data powers the trend charts.

Snapshot Configuration

SettingDefaultDescription
SNAPSHOT_INTERVAL_SECONDS60How often to poll and save metrics
SNAPSHOT_RETENTION_DAYS30How long to keep historical records (auto-purge)

API Endpoint

GET /api/dashboard/history?hours=24&resolution=5m

Returns time-series data aggregated by the requested resolution.

Resolution options: 1m, 5m, 15m, 1h, 6h. Shorter resolutions are only available for narrower time windows to keep responses fast.

Disk Usage Pie Chart

Shows per-server disk usage distribution. Each slice represents a volume server, with labels for node IP and percentage. Color intensity reflects usage level — warmer colors as usage increases.

Volume Growth Trend

An area chart (Recharts) plots total volume count over time. The configurable time range selector (12h, 24h, 7d, 30d) lets you zoom in on recent activity or view long-term growth patterns. Hovering over data points shows exact values at each snapshot interval.

Alert Panel

The alert panel (Ant Design Alert / List components) displays active cluster warnings. Each alert has a severity, timestamp, and actionable controls.

Alert Types

AlertTrigger ConditionDefaultSeverity
Disk UsageAny node exceeds usage percent> 90%Critical
Node OfflineA volume server stops respondingImmediateCritical
Garbage RatioGarbage ratio exceeds threshold> 0.5Warning
ReadOnly VolumesRead-only volume count exceeds max> 3Warning
Disk TemperatureDrive temperature exceeds limit (S.M.A.R.T.)> 55°CWarning
Disk WearSSD wear level exceeds percent (S.M.A.R.T.)> 85%Critical

Alert Lifecycle

1
New Threshold exceeded — alert created with timestamp and deduplication key
2
Acknowledged Operator clicks "Acknowledge" — alert dims, stops re-notifying
3
Resolved Condition clears — alert moves to resolved state, hidden from active panel
Deduplication: The alert engine prevents duplicate alerts for the same issue. If a node exceeds 90% disk usage, only one alert fires. It won't fire again until the condition resolves and reoccurs.

API Endpoints

GET  /api/dashboard/alerts           # List all active alerts
GET  /api/dashboard/alerts?status=all # Include acknowledged & resolved
PUT  /api/dashboard/alerts/{id}/acknowledge  [admin]
GET  /api/dashboard/alerts/config     # Get alert thresholds
PUT  /api/dashboard/alerts/config     # Update thresholds  [admin]

Configurable Thresholds

Thresholds are stored in the runtime_settings SQLite table and managed through the Settings UI. Changes take effect immediately without a server restart.

Operational config vs. infrastructure config: Thresholds and limits live in runtime_settings (changeable at runtime). Hosts, ports, and credentials live in .env (requires server restart). Never mix the two.

Component Health

Every background component (SnapshotService, AlertEngine, DiskHealthService, SseManager) periodically updates a heartbeat in the services_health SQLite table:

ComponentHeartbeat TTL
SnapshotService120s

Check component health via GET /api/health — returns status for all services with their last heartbeat timestamp.
